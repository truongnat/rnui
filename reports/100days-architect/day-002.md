# Day 2/100 — DRY, KISS, YAGNI: Khi nào áp dụng, khi nào không

**Date:** 2026-03-17
**Phase:** 1 — Engineering Foundations

---

## 📚 Lý thuyết cốt lõi

- **DRY (Don't Repeat Yourself):** Mỗi mảnh kiến thức chỉ có một nguồn sự thật duy nhất trong hệ thống. DRY không chỉ về code — nó áp dụng cho config, schema, documentation, business logic.
- **DRY sai cách:** Abstraction quá sớm ("premature DRY") tệ hơn duplication. Hai đoạn code trông giống nhau không có nghĩa là chúng nên merge — wait until the third time (Rule of Three).
- **KISS (Keep It Simple, Stupid):** Simple solution luôn được ưu tiên hơn clever solution. Complexity is a tax — bạn phải trả mỗi lần đọc, debug, maintain.
- **KISS không có nghĩa là naive:** Simple ≠ easy. Tìm được solution đơn giản cho bài toán phức tạp đòi hỏi deep understanding. Einstein: "If you can't explain it simply, you don't understand it well enough."
- **YAGNI (You Aren't Gonna Need It):** Đừng implement feature/abstraction cho tương lai giả định. Cost của wrong abstraction > cost của future refactor.
- **YAGNI và Architect conflict:** Architect hay bị tempt bởi "future-proofing". Nhưng over-engineering hệ thống cho scale không có thực = wasted complexity, slower dev, harder onboarding.
- **Bộ 3 hoạt động cùng nhau:** KISS chọn simple solution, YAGNI tránh thêm thứ không cần, DRY loại bỏ duplication — nhưng theo thứ tự: làm nó work → làm nó right → làm nó fast (not the other way).

## 💡 Ví dụ thực tế bằng TypeScript

### ❌ Vi phạm YAGNI + over-engineering

```typescript
// Quá nhiều abstraction cho bài toán đơn giản
interface IEmailProvider {
  send(options: EmailOptions): Promise<void>;
}

interface IEmailProviderFactory {
  create(type: string): IEmailProvider;
}

class EmailProviderRegistry {
  private providers = new Map<string, () => IEmailProvider>();

  register(type: string, factory: () => IEmailProvider) {
    this.providers.set(type, factory);
  }

  getFactory(): IEmailProviderFactory {
    return {
      create: (type: string) => {
        const factory = this.providers.get(type);
        if (!factory) throw new Error(`Unknown provider: ${type}`);
        return factory();
      },
    };
  }
}

// Thực tế: chỉ dùng SendGrid, không có plan đổi provider
// → YAGNI violation: registry, factory pattern không cần thiết
```

## 🔨 Bài tập thực hành (30-60 phút)

**Task: DRY/KISS Audit on Real Code**

1. Mở một project thực tế bạn đang làm (hoặc dùng project side)
2. Dùng lệnh sau để tìm potential DRY violations:
   ```bash
   # Tìm các function/method signatures tương tự
   grep -rn "function\|const.*=.*=>" src/ --include="*.ts" | sort
   ```
3. Tìm **3 chỗ** có code duplication và phân loại:
   - **Real DRY violation** → nên refactor ngay
   - **Coincidental similarity** → để yên, hai thứ khác nhau
   - **Premature abstraction** → nên un-DRY (yes, đôi khi cần xóa abstraction)
4. Với mỗi chỗ cần refactor, viết test trước rồi mới refactor
5. Commit với message: `refactor: remove accidental complexity in [module]`

**Bonus:** Tìm một chỗ trong codebase đang có abstraction thừa (YAGNI violation) và simplify nó.

## ❓ Câu hỏi phỏng vấn Architect level

1. **"Bạn từng gặp trường hợp áp dụng DRY gây ra vấn đề gì không? Bạn đã xử lý thế nào?"**
   *(Expect: Candidate nhận ra premature abstraction, coupling vô tình giữa các module không liên quan)*

2. **"Khi review code của team, làm sao bạn phân biệt được 'healthy duplication' và 'DRY violation thực sự'?"**
   *(Expect: Hiểu Coincidental vs Essential duplication; hỏi về business meaning của code)*

3. **"Một junior dev muốn build thêm 'abstraction layer để sau này dễ swap database'. Bạn phản hồi thế nào?"**
   *(Expect: YAGNI argument, cost of wrong abstraction, ask về concrete use case; không phải "không bao giờ abstract")*

## 💎 Insight của Architect

> **"The enemy of good code isn't bad code — it's premature abstraction."**

Sau 4 năm code, bạn đã tự nhiên apply DRY. Nhưng khi lên Architect, bạn sẽ review nhiều hơn code của người khác, và cái bẫy lớn nhất là **accept abstraction vì nó "trông pro"**.

Sandi Metz có câu nổi tiếng: *"Duplication is far cheaper than the wrong abstraction."* Một abstraction sai sẽ lan rộng, tạo coupling, và cuối cùng bạn cần rewrite — tốn gấp 10 lần so với để nguyên duplication ban đầu.

**Architect mental model:** Hỏi không phải "code này có lặp không?" mà là "nếu requirement A thay đổi, B có nên thay đổi theo không?" Nếu yes → DRY justified. Nếu no → đó là coincidental similarity, hãy để chúng tồn tại độc lập.
