# Day 5/100 — Design Patterns: Behavioral (Observer, Strategy, Command)
**Date:** 2026-03-20
**Phase:** 1 — Engineering Foundations

---

## 📚 Lý thuyết cốt lõi

- **Behavioral Patterns** giải quyết cách các objects giao tiếp & phân phối trách nhiệm — không phải cấu trúc mà là *hành vi*.
- **Observer**: Định nghĩa mối quan hệ 1-nhiều. Khi một object thay đổi, tất cả dependents tự động được notify. Nền tảng của event-driven và reactive programming (RxJS, EventEmitter).
- **Strategy**: Đóng gói từng thuật toán vào class riêng, cho phép swap runtime. Thay vì if/else dài, bạn inject behavior — tuân thủ Open/Closed Principle.
- **Command**: Đóng gói request thành object. Hỗ trợ undo/redo, queue operations, logging, và transactional behavior.
- Observer khác EventEmitter ở chỗ: Observer là pattern (strong coupling subject↔observer), còn EventEmitter là implementation (loose coupling qua event name).
- Strategy pattern là cốt lõi của Dependency Injection — bạn inject *strategy* thay vì hardcode logic.
- Command pattern xuất hiện khắp nơi: Redux actions, Undo/Redo editors, job queues, CQRS commands.

### Observer Pattern — Price Alert System

```typescript
interface Observer {
  update(price: number): void;
}

interface Subject {
  subscribe(observer: Observer): void;
  unsubscribe(observer: Observer): void;
  notify(): void;
}

class StockPriceTracker implements Subject {
  private observers: Observer[] = [];
  private price: number = 0;

  subscribe(observer: Observer): void {
    this.observers.push(observer);
  }

  unsubscribe(observer: Observer): void {
    this.observers = this.observers.filter(o => o !== observer);
  }

  setPrice(price: number): void {
    this.price = price;
    this.notify();
  }

  notify(): void {
    this.observers.forEach(o => o.update(this.price));
  }
}

class EmailAlert implements Observer {
  update(price: number): void {
    console.log(`📧 Email: TSLA price changed to $${price}`);
  }
}

class PushNotification implements Observer {
  constructor(private threshold: number) {}
  update(price: number): void {
    if (price > this.threshold) {
      console.log(`🔔 Push: TSLA exceeded $${this.threshold}! Now: $${price}`);
    }
  }
}

// Usage
const tracker = new StockPriceTracker();
tracker.subscribe(new EmailAlert());
tracker.subscribe(new PushNotification(200));

tracker.setPrice(195); // Only email fires
tracker.setPrice(205); // Both fire
```

### Strategy Pattern — Payment Processing

```typescript
interface PaymentStrategy {
  pay(amount: number): Promise<{ success: boolean; transactionId: string }>;
}

class StripeStrategy implements PaymentStrategy {
  async pay(amount: number) {
    console.log(`💳 Charging $${amount} via Stripe`);
    return { success: true, transactionId: `stripe_${Date.now()}` };
  }
}

class MoMoStrategy implements PaymentStrategy {
  async pay(amount: number) {
    console.log(`📱 Charging $${amount} via MoMo`);
    return { success: true, transactionId: `momo_${Date.now()}` };
  }
}

class PaymentProcessor {
  constructor(private strategy: PaymentStrategy) {}

  setStrategy(strategy: PaymentStrategy): void {
    this.strategy = strategy;
  }

  async checkout(amount: number) {
    const result = await this.strategy.pay(amount);
    if (result.success) {
      console.log(`✅ Payment confirmed: ${result.transactionId}`);
    }
    return result;
  }
}

// Usage
const processor = new PaymentProcessor(new StripeStrategy());
await processor.checkout(99.99);

processor.setStrategy(new MoMoStrategy()); // Swap at runtime
await processor.checkout(50000);
```

### Command Pattern — Text Editor với Undo/Redo

```typescript
interface Command {
  execute(): void;
  undo(): void;
}

class TextEditor {
  private content: string = '';
  private history: Command[] = [];
  private undoneStack: Command[] = [];

  executeCommand(command: Command): void {
    command.execute();
    this.history.push(command);
    this.undoneStack = []; // Clear redo stack
  }

  undo(): void {
    const command = this.history.pop();
    if (command) {
      command.undo();
      this.undoneStack.push(command);
    }
  }

  redo(): void {
    const command = this.undoneStack.pop();
    if (command) {
      command.execute();
      this.history.push(command);
    }
  }

  getContent(): string { return this.content; }
  setContent(text: string): void { this.content = text; }
}

class InsertTextCommand implements Command {
  private previousContent: string = '';

  constructor(
    private editor: TextEditor,
    private text: string
  ) {}

  execute(): void {
    this.previousContent = this.editor.getContent();
    this.editor.setContent(this.previousContent + this.text);
    console.log(`Content: "${this.editor.getContent()}"`);
  }

  undo(): void {
    this.editor.setContent(this.previousContent);
    console.log(`Undo! Content: "${this.editor.getContent()}"`);
  }
}

// Usage
const editor = new TextEditor();
editor.executeCommand(new InsertTextCommand(editor, 'Hello'));
editor.executeCommand(new InsertTextCommand(editor, ' World'));
editor.undo(); // Undo " World"
editor.redo(); // Redo " World"
```

## 🔨 Bài tập thực hành

**Task: Xây dựng một Notification System với 3 patterns kết hợp**

Tạo một hệ thống notification cho e-commerce platform:

1. **Observer**: `OrderService` phát events khi order status thay đổi (PENDING → CONFIRMED → SHIPPED → DELIVERED)
2. **Strategy**: Có 3 notification strategies: `EmailNotifier`, `SMSNotifier`, `PushNotifier` — user có thể chọn combination
3. **Command**: Wrap mỗi notification action thành Command để có thể retry khi fail, và log audit trail

**Acceptance criteria:**
- Khi order được confirm, gửi notification qua tất cả channels mà user đã đăng ký
- Failed notifications có thể retry mà không cần duplicate code
- Có thể xem audit log của tất cả notifications đã gửi

**Gợi ý file structure:**
```
src/
  patterns/
    observer/  → OrderEventEmitter, NotificationObserver
    strategy/  → EmailStrategy, SMSStrategy, PushStrategy
    command/   → SendNotificationCommand, NotificationQueue
  services/
    OrderService.ts
    NotificationOrchestrator.ts
```

## ❓ Câu hỏi phỏng vấn (Architect level)

**Q1:** Trong một hệ thống microservices, bạn sẽ chọn Observer pattern (synchronous) hay Event-Driven Architecture (async message queue) cho inter-service communication? Trade-off là gì?

> *Gợi ý:* Observer = tight coupling, đồng bộ, đơn giản. EDA = loose coupling, async, fault-tolerant nhưng phức tạp hơn (eventual consistency, message ordering).

**Q2:** Strategy pattern và Dependency Injection khác nhau như thế nào? Khi nào bạn dùng cái nào?

> *Gợi ý:* Strategy = behavioral intent (swap algorithm), DI = structural concern (manage dependencies). Strategy thường được inject, nhưng không phải tất cả DI là Strategy.

**Q3:** Command pattern trong Redux và CQRS có gì giống nhau? Điểm nào Redux không làm được so với full CQRS implementation?

> *Gợi ý:* Cả hai đóng gói intent thành object. Redux thiếu command routing, handler separation, và event sourcing baked-in.

## 💎 Insight của Architect

**"Behavioral patterns là ngôn ngữ của collaboration."**

Khi bạn review code và thấy một đống if/else kiểm tra type để quyết định hành vi → đó là dấu hiệu cần Strategy. Khi bạn thấy tight coupling giữa producer và consumer → Observer hoặc Events. Khi bạn cần audit, undo, hoặc retry → Command.

Điều thực sự quan trọng: **đừng áp pattern vào code cho có**. Một Architect giỏi biết khi nào *không* dùng pattern. Strategy overkill cho 2 conditions. Observer thừa nếu chỉ có 1 listener không bao giờ thay đổi. Sự đơn giản là kiến trúc tốt nhất — pattern chỉ justified khi nó giải quyết một vấn đề thực sự.

*"Premature pattern application is the root of all accidental complexity."*

## 🧑‍💼 Day 5/100 — Leader Skills: Viết Technical Documentation

### 🧠 Concept cốt lõi

1. **Docs là sản phẩm, không phải phụ lục** — Documentation tốt = team không cần hỏi bạn mỗi ngày. Architect viết doc để *nhân bản* kiến thức của mình.
2. **Audience-first writing** — Luôn hỏi: "Ai sẽ đọc cái này?" Dev mới? PM? Stakeholder? Mỗi đối tượng cần level of detail khác nhau.
3. **Pyramid structure** — Bắt đầu bằng *kết luận/TL;DR*, sau đó mới chi tiết. Người bận chỉ đọc đầu, người cần hiểu sâu đọc tiếp.
4. **Living docs > Perfect docs** — Một doc tốt nhưng outdated còn nguy hiểm hơn không có doc. Gắn doc với code (ví dụ: ADR trong repo).
5. **Quyết định > Mô tả** — Doc kỹ thuật giỏi không chỉ ghi "hệ thống làm gì" mà còn ghi "tại sao chọn cách này, đã cân nhắc gì, đánh đổi là gì".

---

### 📖 Tình huống thực tế

**Scenario:** Bạn vừa xây xong hệ thống notification service dùng Redis Pub/Sub thay vì Kafka. PM hỏi "tại sao không dùng Kafka?", dev mới vào team cũng không hiểu flow. Bạn mất 30 phút giải thích lần thứ 3 trong tuần.

**Giải pháp:** Viết một **ADR (Architecture Decision Record)** cho quyết định này. Chỉ mất 20 phút viết, nhưng tiết kiệm hàng giờ giải thích sau này. ADR gồm: Context → Decision → Consequences (positive/negative).

---

### ✍️ Bài tập thực hành (15–30 phút)

Chọn **1 quyết định kỹ thuật bạn đã làm gần đây** (dùng library X thay Y, chọn cấu trúc folder, cách handle authentication...) và viết ADR theo template:

```markdown
# ADR-001: [Tên quyết định]

## Status: Accepted

## Context
[Vấn đề/bối cảnh bạn đối mặt là gì?]

## Decision
[Bạn đã chọn gì?]

## Rationale
[Tại sao chọn vậy? Đã cân nhắc những gì?]

## Consequences
✅ [Lợi ích]
⚠️ [Đánh đổi / rủi ro]
```

Lưu vào `docs/decisions/ADR-001.md` trong project hiện tại.
