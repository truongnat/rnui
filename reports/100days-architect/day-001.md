# Day 1/100 — SOLID Principles
**Date:** 2026-03-17
**Topic:** SOLID Principles — deep dive với JS examples

---

## 📚 Lý thuyết cốt lõi

- **S — Single Responsibility Principle (SRP):** Mỗi class/module chỉ có một lý do để thay đổi. Không phải "làm một việc" — mà là "chịu trách nhiệm với một actor duy nhất".
- **O — Open/Closed Principle (OCP):** Open for extension, closed for modification. Dùng abstraction (interface, base class) để thêm behavior mới mà không sửa code cũ.
- **L — Liskov Substitution Principle (LSP):** Subtype phải thay thế được supertype mà không phá vỡ behavior. Vi phạm LSP thường lộ ra khi dùng `instanceof` checks hoặc override throw error.
- **I — Interface Segregation Principle (ISP):** Nhiều interface nhỏ chuyên biệt tốt hơn một interface lớn "fat". Client không nên phụ thuộc vào method mà nó không dùng.
- **D — Dependency Inversion Principle (DIP):** High-level modules không depend vào low-level modules — cả hai depend vào abstractions. Đây là nền tảng của Dependency Injection.
- **Mối liên hệ:** SOLID không phải 5 quy tắc độc lập — chúng liên kết: DIP enable OCP, SRP làm ISP dễ hơn, LSP giúp OCP thực sự hoạt động.
- **Khi nào không cần áp dụng cứng nhắc:** Script nhỏ, prototype, code throw-away — over-engineering SOLID vào code ít thay đổi là lãng phí. Architect biết khi nào trade-off hợp lý.

## 💡 Ví dụ thực tế bằng TypeScript

```typescript
// ❌ Vi phạm SRP + DIP
class UserService {
  async createUser(data: { name: string; email: string }) {
    // Business logic + DB + Email + Logging tất cả trong 1 class
    const user = await db.query(`INSERT INTO users...`);
    await nodemailer.sendMail({ to: data.email, subject: 'Welcome!' });
    console.log(`User created: ${data.email}`);
    return user;
  }
}

// ✅ Áp dụng SRP + DIP + ISP
interface IUserRepository {
  save(user: User): Promise<User>;
}

interface IEmailService {
  sendWelcome(email: string): Promise<void>;
}

interface ILogger {
  log(message: string): void;
}

class UserService {
  constructor(
    private readonly userRepo: IUserRepository,   // DIP: depend on abstraction
    private readonly emailService: IEmailService,
    private readonly logger: ILogger,
  ) {}

  async createUser(data: { name: string; email: string }): Promise<User> {
    const user = await this.userRepo.save(new User(data));  // SRP: chỉ orchestrate
    await this.emailService.sendWelcome(data.email);
    this.logger.log(`User created: ${data.email}`);
    return user;
  }
}

// ✅ OCP — thêm discount type mới không sửa code cũ
interface DiscountStrategy {
  calculate(price: number): number;
}

class PercentageDiscount implements DiscountStrategy {
  constructor(private percent: number) {}
  calculate(price: number) { return price * (1 - this.percent / 100); }
}

class FixedDiscount implements DiscountStrategy {
  constructor(private amount: number) {}
  calculate(price: number) { return price - this.amount; }
}

// Thêm FlashSaleDiscount → không cần sửa OrderService!
class OrderService {
  applyDiscount(price: number, strategy: DiscountStrategy): number {
    return strategy.calculate(price);
  }
}

// ✅ LSP violation example
class Bird {
  fly(): void { console.log('flying'); }
}

class Penguin extends Bird {
  fly(): void { throw new Error('Penguins cannot fly!'); } // ❌ LSP vi phạm!
}

// Fix: tách interface
interface Flyable { fly(): void; }
interface Swimmable { swim(): void; }

class Eagle implements Flyable {
  fly() { console.log('soaring high'); }
}

class Penguin implements Swimmable {
  swim() { console.log('swimming fast'); }
}
```

## 🔨 Bài tập thực hành

**Task: Refactor một UserController thực tế**

Lấy một controller trong project hiện tại của bạn (hoặc tạo mới) có:
- Validation logic
- Business logic
- Database calls
- Response formatting

**Yêu cầu:**
1. Identify tất cả SRP violations trong controller đó
2. Tách ra các layers: Controller → Service → Repository
3. Định nghĩa interfaces cho Service và Repository (DIP)
4. Viết unit test cho Service layer bằng cách mock Repository
5. Document lại: "Trước refactor, class này có mấy lý do để thay đổi?"

**Kết quả mong đợi:** Service layer có thể test hoàn toàn độc lập với database.

## ❓ Câu hỏi phỏng vấn Architect level

1. **"Trong microservices, SRP áp dụng ở level nào — class, service, hay cả hai? Và làm sao bạn xác định boundary của một microservice theo SRP?"**
   → Gợi ý: Liên kết với DDD Bounded Context, Conway's Law

2. **"OCP nói 'đóng với modification' — nhưng khi có bug fix, bạn vẫn phải sửa code. Bạn reconcile điều này như thế nào? Và khi nào OCP trở thành over-engineering?"**
   → Gợi ý: OCP áp dụng cho *expected extension points*, không phải mọi thứ

3. **"Bạn đang review code của một junior dev: họ tạo một AbstractBaseService với 15 methods. Bạn nhận ra điều gì và sẽ nói gì với họ?"**
   → Gợi ý: ISP violation, God class, prefer composition over inheritance

## 💎 Insight của Architect

**"SOLID là ngôn ngữ chung, không phải luật lệ."**

Sau 4 năm code, bạn đã áp dụng SOLID intuitive mà không biết tên. Giờ là lúc formalize nó. Giá trị thực của SOLID với Architect không phải là "viết code đúng" — mà là **dùng nó như shared vocabulary khi review code và design system**.

Khi bạn nói "class này vi phạm SRP" trong code review, cả team hiểu ngay — không cần giải thích dài. Khi design microservice boundary, DIP giúp bạn argue tại sao service A không nên call thẳng database của service B.

**Trade-off cần nhớ:** Áp dụng SOLID cứng nhắc trong codebase nhỏ tạo ra boilerplate không cần thiết. Architect giỏi biết *when* to apply, không chỉ *how* to apply.

## 🧑‍💼 Day 1/100 — Leader Skills: Kỹ năng Giao tiếp Kỹ thuật (Technical Communication)
*#100DaysLeader | 2026-03-17*

### 🧠 Concept cốt lõi

1. **Giao tiếp kỹ thuật ≠ nói nhiều** — Tech Lead giỏi là người biết *chọn lọc* thông tin: đúng người, đúng lúc, đúng mức độ chi tiết.
2. **Audience-first mindset** — Trước khi nói/viết, luôn hỏi: *"Người này cần biết gì?"* Dev cần context kỹ thuật, PM cần timeline & risk, CTO cần impact & trade-off.
3. **The Pyramid Principle** — Bắt đầu bằng kết luận, sau đó mới giải thích lý do. Đừng kể chuyện dài mới đưa ra điểm chính.
4. **Viết tốt = tư duy tốt** — Nếu bạn không thể giải thích rõ ràng bằng văn bản, bạn chưa thực sự hiểu vấn đề.
5. **Async-first communication** — Trong môi trường remote/hybrid, văn bản rõ ràng tiết kiệm hàng chục cuộc họp không cần thiết.

---

### 📖 Tình huống thực tế

**Scenario:** Team bạn vừa hoàn thành migration từ REST API sang GraphQL. PM hỏi: *"Tuần này mình đã làm được gì?"*

❌ **Sai:** *"Em đã refactor toàn bộ resolver, setup DataLoader để tránh N+1 problem, viết schema cho 12 entity, config Apollo Server với persisted queries..."*

✅ **Đúng:** *"Mình đã hoàn thành migration API sang GraphQL. Kết quả: response time giảm 40%, số lượng request xuống 3x so với trước. Tuần tới sẽ deploy lên staging để test."*

**Tại sao?** PM cần biết *impact*, không cần biết bạn làm *cái gì*. Kỹ thuật là phương tiện, không phải mục tiêu.

Gửi thật hoặc lưu draft — quan trọng là *thực hành viết*.

## 🧑‍💼 Day 2/100 — Leader Skills: Technical Writing & Documentation

### 🧠 Concept cốt lõi
1. **Doc là code** — Tài liệu kỹ thuật tệ = technical debt vô hình. Một PR thiếu context làm reviewer mất 20 phút, nhân với 100 PR/tháng = 33+ giờ lãng phí.
2. **Audience-first writing** — Luôn hỏi: "Người đọc này là ai? Junior dev? PM? CTO?" Mỗi audience cần level of detail khác nhau.
3. **Structure beats prose** — Bullet points, headers, code blocks > đoạn văn dài. Engineers đọc lướt, không đọc đủ.
4. **Decision records (ADR)** — Mọi quyết định kiến trúc cần ghi lại *lý do*, không chỉ *kết quả*. 6 tháng sau bạn sẽ cảm ơn chính mình.
5. **Living docs** — Tài liệu không được cập nhật = tài liệu nói dối. Gắn doc review vào sprint cycle.

### 📖 Tình huống thực tế
> **Scenario:** Bạn vừa hoàn thành migration từ REST sang GraphQL cho một service. PM hỏi "tại sao chúng ta làm vậy?" — Junior dev mới join hỏi "làm thế nào để dùng API mới?" — CTO hỏi "impact là gì?"

Ba câu hỏi, ba người đọc, ba loại tài liệu:
- **ADR** cho PM/CTO: "Chúng ta chọn GraphQL vì X, Y, Z. Trade-off là A, B."
- **API Guide** cho dev: Endpoint, query examples, error handling
- **Executive Summary** cho CTO: 2-3 dòng impact (performance +30%, bundle size -20%)

Nếu không có 3 tài liệu này, bạn sẽ trả lời 3 câu hỏi đó lặp đi lặp lại mỗi tuần.

### ✍️ Bài tập thực hành (15-20 phút)
**Task:** Chọn 1 technical decision bạn đã làm trong 2 tuần gần đây (chọn lib, refactor một module, thay đổi pattern...) và viết **ADR ngắn** theo template dưới đây. Lưu vào `docs/adr/` trong project của bạn.

### 💬 Mẫu câu / Template — Architecture Decision Record (ADR)

```markdown
# ADR-001: [Tên quyết định]

**Date:** YYYY-MM-DD
**Status:** Accepted | Proposed | Deprecated

## Context
[Vấn đề gì đang xảy ra? Tại sao cần đưa ra quyết định?]

## Decision
[Chúng ta quyết định làm gì?]

## Rationale
- ✅ [Lý do 1]
- ✅ [Lý do 2]
- ❌ [Trade-off / nhược điểm chấp nhận được]

## Alternatives Considered
- Option A: ... — Rejected vì ...
- Option B: ... — Rejected vì ...

## Consequences
- [Impact ngắn hạn]
- [Impact dài hạn]
```

### 💎 Insight của Leader
> *"Một Tech Lead giỏi không phải là người code nhiều nhất trong team — mà là người làm cho cả team code hiệu quả hơn. Documentation là đòn bẩy đó. Mỗi giờ bạn bỏ ra viết doc tốt tiết kiệm 10 giờ onboarding, 5 giờ họp giải thích, và vô số lần 'sao code lại như này nhỉ?' lúc 11 giờ đêm."*
>
> Bắt đầu nhỏ: commit vào việc viết **1 ADR mỗi sprint**. Sau 3 tháng, codebase của bạn sẽ có một lịch sử quyết định mà không team nào cũng có được.
