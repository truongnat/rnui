# Day 4/100 — Design Patterns: Structural (Adapter, Decorator, Proxy)
Date: 2026-03-19

## 📚 Lý thuyết cốt lõi

- **Structural Patterns** giải quyết bài toán *tổ chức class/object* thành cấu trúc lớn hơn, giữ tính linh hoạt và hiệu quả
- **Adapter Pattern**: Biến interface không tương thích thành interface mà client mong đợi — như "ổ cắm đa năng" trong lập trình. Dùng khi tích hợp third-party lib hoặc legacy code
- **Decorator Pattern**: Thêm hành vi vào object *tại runtime* mà không thay đổi class gốc — tuân thủ Open/Closed Principle. Khác với inheritance: composable & stackable
- **Proxy Pattern**: Tạo object đại diện kiểm soát truy cập vào object thật — dùng cho lazy loading, caching, access control, logging
- Cả 3 pattern đều wrap một object, nhưng **mục đích khác nhau**: Adapter thay đổi *interface*, Decorator thêm *behaviour*, Proxy kiểm soát *access*
- Trong Node.js/TypeScript, Decorator gắn liền với `@decorators` (NestJS dùng rất nhiều), Proxy gắn liền với ES6 `Proxy` built-in
- Architect dùng Structural Patterns để thiết kế **seam points** — nơi có thể thay thế implementation mà không ảnh hưởng phần còn lại của hệ thống

## 💡 Ví dụ thực tế bằng TypeScript

```typescript
// ============================================
// ADAPTER PATTERN
// ============================================
// Scenario: Bạn có legacy payment system dùng XML,
// nhưng app mới dùng JSON. Adapter giải quyết không thụt.

interface ModernPaymentGateway {
  processPayment(amount: number, currency: string): Promise<{ success: boolean; txId: string }>;
}

// Legacy system - không thể sửa
class LegacyXMLPaymentSystem {
  executeTransaction(xmlPayload: string): string {
    // trả về XML response
    return `<transaction><status>OK</status><id>TX-${Date.now()}</id></transaction>`;
  }
}

// Adapter: wrap legacy, expose modern interface
class PaymentAdapter implements ModernPaymentGateway {
  constructor(private legacy: LegacyXMLPaymentSystem) {}

  async processPayment(amount: number, currency: string) {
    const xml = `<payment><amount>${amount}</amount><currency>${currency}</currency></payment>`;
    const xmlResponse = this.legacy.executeTransaction(xml);
    
    // parse XML -> JSON (simplified)
    const success = xmlResponse.includes('<status>OK</status>');
    const txId = xmlResponse.match(/<id>(.*?)<\/id>/)?.[1] ?? '';
    
    return { success, txId };
  }
}

// ============================================
// DECORATOR PATTERN
// ============================================
// Scenario: Thêm logging, caching vào service mà không sửa business logic

interface UserService {
  getUser(id: string): Promise<{ id: string; name: string }>;
}

class UserServiceImpl implements UserService {
  async getUser(id: string) {
    // DB call...
    return { id, name: 'Truong DQ' };
  }
}

// Logging Decorator
class LoggingUserService implements UserService {
  constructor(private inner: UserService) {}

  async getUser(id: string) {
    console.log(`[LOG] getUser called with id=${id}`);
    const result = await this.inner.getUser(id);
    console.log(`[LOG] getUser returned:`, result);
    return result;
  }
}

// Caching Decorator
class CachingUserService implements UserService {
  private cache = new Map<string, { id: string; name: string }>();

  constructor(private inner: UserService) {}

  async getUser(id: string) {
    if (this.cache.has(id)) {
      console.log(`[CACHE] HIT for id=${id}`);
      return this.cache.get(id)!;
    }
    const result = await this.inner.getUser(id);
    this.cache.set(id, result);
    return result;
  }
}

// Composable: stack decorators
const userService: UserService = new CachingUserService(
  new LoggingUserService(
    new UserServiceImpl()
  )
);

// ============================================
// PROXY PATTERN (ES6 Proxy)
// ============================================
// Scenario: Kiểm soát access vào config object — chặn ghi đè key quan trọng

const appConfig = {
  dbUrl: 'postgres://localhost:5432/mydb',
  jwtSecret: 'super-secret',
  maxRetries: 3,
};

const PROTECTED_KEYS = ['dbUrl', 'jwtSecret'];

const safeConfig = new Proxy(appConfig, {
  set(target, key: string, value) {
    if (PROTECTED_KEYS.includes(key)) {
      throw new Error(`[SECURITY] Cannot override protected config key: "${key}"`);
    }
    (target as any)[key] = value;
    return true;
  },
  get(target, key: string) {
    console.log(`[ACCESS] Config key accessed: "${key}"`);
    return (target as any)[key];
  }
});

// safeConfig.jwtSecret = 'hacked'; // throws Error!
// safeConfig.maxRetries = 5; // OK
```

Tạo file `http-client.ts` với các yêu cầu:

3. Viết `RetryDecorator` — tự động retry tối đa 3 lần nếu request fail
4. Viết `CacheDecorator` — cache response theo URL trong 60 giây
5. Viết `LoggingDecorator` — log mọi request và response time
6. Compose: `new CacheDecorator(new RetryDecorator(new LoggingDecorator(new FetchHttpClient())))`
7. **Bonus**: Viết `CircuitBreakerDecorator` — tự động "mở circuit" nếu fail quá 5 lần liên tiếp

## 💎 Insight của Architect

> **"Structural patterns là nghệ thuật vẽ đường biên."**
>
> Khi bạn còn là junior, bạn hỏi "pattern này hoạt động thế nào?". Khi là Architect, bạn hỏi "pattern này tạo ra seam point ở đâu, và ai sẽ own cái seam đó?".
>
> Adapter pattern không chỉ là "technical glue" — nó là *anti-corruption layer*. Khi bạn wrap Stripe SDK bằng một `PaymentGateway` interface, bạn đang nói: "Business domain của tôi không phụ thuộc vào Stripe. Stripe chỉ là một implementation detail." Đây là tư duy Architect: thiết kế hệ thống để **thay đổi vendor không cần rewrite business logic**.
>
> Bài học từ thực tế: 80% những lần "migration nightmare" mà tôi thấy đều do team không đặt Adapter layer từ đầu. Một interface + một Adapter tốn 30 phút hôm nay, tiết kiệm 3 tuần migration sau này.

## 🧑‍💼 Day 4/100 — Leader Skills: Viết Technical Document Hiệu Quả

### 🧠 Concept cốt lõi

1. **Document là sản phẩm, không phải ghi chú** — Một technical doc tốt phải có audience rõ ràng, mục đích rõ ràng, và dễ scan nhanh (headers, bullets, diagrams).
2. **Pyramid Principle** — Bắt đầu bằng kết luận/tóm tắt, sau đó mới giải thích chi tiết. Người đọc bận rộn không đọc từ đầu đến cuối.
3. **4 loại doc cốt lõi trong tech team:** ADR (Architecture Decision Record), RFC (Request For Comments), Runbook, và Postmortem. Mỗi loại có mục đích riêng biệt.
4. **"Living documents" vs "Dead documents"** — Doc không được cập nhật = doc độc hại. Gắn owner + review date cho mỗi document quan trọng.
5. **Diagrams > Paragraphs** — Một sơ đồ C4 hoặc sequence diagram thay được 3 đoạn văn. Dùng Mermaid/draw.io để embed trực tiếp vào Markdown.

### 📖 Tình huống thực tế

**Scenario:** Team bạn vừa quyết định chuyển từ monolith sang microservices cho một module thanh toán. PM hỏi: *"Tại sao chúng ta làm vậy? Rủi ro là gì?"* Dev junior mới join hỏi: *"Kiến trúc cũ thế nào? Tôi bắt đầu từ đâu?"*

→ **Không có ADR**, bạn phải giải thích lại từ đầu cho từng người — mỗi lần mất 30 phút.
→ **Có ADR**, bạn gửi 1 link: quyết định, context, alternatives đã cân nhắc, hệ quả. Xong.

Đây là sự khác biệt giữa team "chạy bằng miệng" và team "chạy bằng hệ thống".

### ✍️ Bài tập thực hành (15–30 phút)

**Task:** Viết 1 ADR (Architecture Decision Record) mini cho 1 quyết định kỹ thuật bạn đã làm gần đây trong dự án.

Dùng template MADR (Markdown Architectural Decision Records):

```markdown
# ADR-001: [Tên quyết định]

## Status
Accepted | Deprecated | Superseded by ADR-XXX

## Context
[Vấn đề bạn đang giải quyết là gì? Tại sao cần đưa ra quyết định này?]

## Decision
[Bạn đã quyết định làm gì?]

## Alternatives Considered
- Option A: ... (lý do không chọn)
- Option B: ... (lý do không chọn)

## Consequences
- ✅ Positive: ...
- ⚠️ Negative/Tradeoffs: ...

## Date
2026-03-19 | Author: Truong DQ
```

→ Lưu vào `docs/adr/` trong repo. Làm thói quen mỗi khi có quyết định tech quan trọng.
