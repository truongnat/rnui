# Day 6/100 — Clean Architecture: Layers & Dependency Rule
**Date:** 2026-03-21
**Topic:** Clean Architecture — layers & dependency rule

---

## 📚 Lý thuyết cốt lõi

- **Clean Architecture** (Robert C. Martin) tổ chức code thành các vòng tròn đồng tâm: Entities → Use Cases → Interface Adapters → Frameworks & Drivers — mỗi layer chỉ phụ thuộc vào layer bên trong.
- **The Dependency Rule:** Source code dependencies chỉ được trỏ vào *trong* (inward). Outer layers biết inner layers, nhưng inner layers KHÔNG bao giờ biết outer layers.
- **Entities (Domain Layer):** Business rules thuần túy, không phụ thuộc framework nào. Tồn tại lâu nhất, thay đổi ít nhất.
- **Use Cases (Application Layer):** Orchestrate luồng dữ liệu đến/từ Entities, implement business logic cụ thể của application. Không biết gì về HTTP, DB, UI.
- **Interface Adapters:** Controllers, Presenters, Gateways — chuyển đổi dữ liệu từ format của Use Cases sang format của framework (và ngược lại).
- **Frameworks & Drivers (Infrastructure Layer):** Express, NestJS, TypeORM, PostgreSQL — chi tiết triển khai. Dễ thay đổi nhất.
- **Lợi ích thực tế:** Testability cao (test Use Cases mà không cần DB), flexibility (swap database/framework mà không đụng business logic), maintainability (dễ hiểu, dễ onboard team mới).

## 💡 Ví dụ thực tế bằng TypeScript

```typescript
// ============================
// LAYER 1: ENTITIES (Domain)
// ============================
// Không import gì từ bên ngoài
export class User {
  constructor(
    public readonly id: string,
    public readonly email: string,
    private readonly passwordHash: string
  ) {}

  isValidEmail(): boolean {
    return /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(this.email);
  }
}

// ============================
// LAYER 2: USE CASES (Application)
// ============================
// Định nghĩa interface (port) — KHÔNG import implementation
export interface UserRepository {
  findByEmail(email: string): Promise<User | null>;
  save(user: User): Promise<void>;
}

export interface PasswordHasher {
  hash(plain: string): Promise<string>;
}

// Use Case: thuần business logic
export class RegisterUserUseCase {
  constructor(
    private readonly userRepo: UserRepository,
    private readonly hasher: PasswordHasher
  ) {}

  async execute(email: string, password: string): Promise<User> {
    const existing = await this.userRepo.findByEmail(email);
    if (existing) throw new Error('Email already registered');

    const tempUser = new User('', email, '');
    if (!tempUser.isValidEmail()) throw new Error('Invalid email format');

    const passwordHash = await this.hasher.hash(password);
    const user = new User(crypto.randomUUID(), email, passwordHash);
    await this.userRepo.save(user);
    return user;
  }
}

// ============================
// LAYER 3: INTERFACE ADAPTERS
// ============================
// Controller biết Use Case nhưng không biết DB
export class RegisterUserController {
  constructor(private readonly useCase: RegisterUserUseCase) {}

  async handle(req: { body: { email: string; password: string } }) {
    try {
      const user = await this.useCase.execute(req.body.email, req.body.password);
      return { status: 201, body: { id: user.id, email: user.email } };
    } catch (err: any) {
      return { status: 400, body: { error: err.message } };
    }
  }
}

// ============================
// LAYER 4: INFRASTRUCTURE
// ============================
// Implement interface từ Use Case layer — dependency đi vào trong
import bcrypt from 'bcrypt';

export class BcryptPasswordHasher implements PasswordHasher {
  async hash(plain: string): Promise<string> {
    return bcrypt.hash(plain, 10);
  }
}

// PostgreSQL implementation của UserRepository
export class PostgresUserRepository implements UserRepository {
  constructor(private readonly db: any) {} // db = pg Pool

  async findByEmail(email: string): Promise<User | null> {
    const result = await this.db.query('SELECT * FROM users WHERE email = $1', [email]);
    if (!result.rows[0]) return null;
    const row = result.rows[0];
    return new User(row.id, row.email, row.password_hash);
  }

  async save(user: User): Promise<void> {
    await this.db.query(
      'INSERT INTO users (id, email, password_hash) VALUES ($1, $2, $3)',
      [user.id, user.email, (user as any).passwordHash]
    );
  }
}

// ============================
// COMPOSITION ROOT (main.ts)
// ============================
// Đây là nơi DUY NHẤT wire dependencies lại với nhau
const db = new Pool({ connectionString: process.env.DATABASE_URL });
const hasher = new BcryptPasswordHasher();
const userRepo = new PostgresUserRepository(db);
const registerUseCase = new RegisterUserUseCase(userRepo, hasher);
const registerController = new RegisterUserController(registerUseCase);
```

**Key insight từ code trên:** `RegisterUserUseCase` không hề import bcrypt hay pg. Bạn có thể test nó với mock implementations — không cần database thật.

## ❓ Câu hỏi phỏng vấn Architect level

**Q1:** "Trong Clean Architecture, làm sao bạn handle cross-cutting concerns như logging và authentication mà không vi phạm Dependency Rule?"

*Gợi ý trả lời:* Dùng Decorator pattern hoặc Middleware ở Interface Adapter layer. Logging nằm ở infrastructure, nhưng nếu Use Case cần log, define `Logger` interface trong Application layer và inject implementation từ Infrastructure.

**Q2:** "Khi nào Clean Architecture là overkill và bạn sẽ chọn gì thay thế?"

*Gợi ý trả lời:* Clean Arch overkill với CRUD apps đơn giản, MVP, scripts. Đối với projects < 3 tháng hoặc team 1-2 người, Simple Layered Architecture (controllers → services → repos) đủ dùng. Clean Arch shine khi domain logic phức tạp, cần swap infrastructure, hoặc team > 5 người.

**Q3:** "Làm thế nào bạn migrate một codebase monolith legacy sang Clean Architecture mà không rewrite toàn bộ?"

*Gợi ý trả lời:* Strangler Fig Pattern — bọc legacy code dần dần. Bắt đầu từ new features dùng Clean Arch, dần extract Use Cases từ fat controllers. Tạo Anti-Corruption Layer để cô lập legacy code. Không bao giờ "big bang rewrite".

## 💎 Insight của Architect

> **"Frameworks are details, not decisions."**

Đây là shift tư duy quan trọng nhất khi lên Architect. Dev thường chọn framework rồi thiết kế hệ thống *xung quanh* nó — đây là sai lầm. Clean Architecture dạy bạn thiết kế business logic *trước*, framework chỉ là plugin thay được.

Trong thực tế, bạn có thể serve cùng một Use Case qua HTTP API hôm nay, qua GraphQL tháng sau, và qua gRPC năm sau — mà không cần sửa 1 dòng trong domain layer.

Khi review code của team, Architect cần hỏi: *"Nếu tôi thay Express bằng Fastify, bao nhiêu file phải thay đổi?"* — nếu câu trả lời là hàng chục files, architecture đang có vấn đề.

## 🧑‍💼 Day 6/100 — Leader Skills: Technical Documentation

### 🧠 Concept cốt lõi

1. **Docs là code thứ hai** — Một Architect giỏi không chỉ thiết kế hệ thống, mà còn phải truyền đạt thiết kế đó rõ ràng cho team. Nếu chỉ bạn hiểu, thì khi bạn vắng mặt, cả team tê liệt.
2. **Docs = Quyết định có thể audit** — Mỗi tài liệu kỹ thuật nên ghi lại WHY (lý do quyết định), không chỉ WHAT (cái đã làm). ADR (Architecture Decision Record) là công cụ tiêu chuẩn để làm điều này.
3. **Level hóa audience** — Viết cho 3 đối tượng khác nhau: (a) Developer cần implement → chi tiết kỹ thuật; (b) Tech Lead review → trade-offs, risks; (c) Manager/Stakeholder → business impact, timeline.
4. **Keep it living, not dead** — Docs không được cập nhật còn tệ hơn không có docs, vì nó dẫn đường sai. Gắn docs review vào quy trình PR/sprint review.
5. **Diagram > Words** — C4 Model (Context → Container → Component → Code) là chuẩn vàng để visualize kiến trúc. Một diagram đúng chỗ thay 500 từ mô tả.

### 📖 Tình huống thực tế

**Scenario:** Team bạn vừa build xong một microservice mới `notification-service`. PM hỏi: *"Service này hoạt động thế nào? Tại sao không dùng Firebase FCM có sẵn?"*

Bạn biết câu trả lời, nhưng khi mở codebase ra để giải thích... mất 20 phút tìm entry point, config, và flow. Nếu bạn đã viết docs trước khi merge PR:

- **README.md** → Giải thích service làm gì, cách chạy local
- **ADR-001-notification-service.md** → Tại sao không dùng FCM: vì cần multi-channel (email + push + SMS), vendor lock-in risk, và cost control
- **Sequence Diagram** → Flow từ event → queue → notification delivery

Kết quả: PM đọc trong 5 phút, team mới onboard trong 30 phút thay vì 3 ngày.

### ✍️ Bài tập thực hành (15–30 phút)

**Task:** Viết 1 ADR (Architecture Decision Record) cho một quyết định kỹ thuật bạn đã từng làm — dù nhỏ cũng được.

**Template ADR:**
```markdown
# ADR-XXX: [Tên quyết định]

## Status
[Proposed | Accepted | Deprecated | Superseded]

## Context
[Vấn đề gì đang xảy ra? Tại sao cần đưa ra quyết định này?]

## Decision
[Chúng ta quyết định làm gì?]

## Consequences
**Positive:**
- ...

**Negative / Trade-offs:**
- ...

## Alternatives Considered
- Option A: ...
- Option B: ...
```

**Ví dụ topic để viết:** "Tại sao chọn PostgreSQL thay vì MongoDB?", "Tại sao dùng monorepo?", "Tại sao implement JWT thay vì session-based auth?"

### 💎 Insight của Leader

> *"The most senior engineers I've worked with spend 30–40% of their time writing — docs, proposals, post-mortems, ADRs. They understood that code is temporary, but decisions documented outlast both the code and the coder."*

**Thực tế từ Tech Lead:** Khi một Architect rời công ty, nếu không có docs, knowledge chết theo. Nhưng nếu có hệ thống ADRs tốt, người mới có thể hiểu *tại sao* hệ thống được thiết kế như vậy — không phải chỉ *cái gì* đang chạy. Đó là sự khác biệt giữa một Senior Developer và một Architect thực sự: **bạn để lại wisdom, không chỉ code.**

Hành động cụ thể: Từ hôm nay, khi commit bất kỳ thay đổi kiến trúc nào, thêm 1 ADR file vào repo. Sau 3 tháng, bạn có một knowledge base vô giá.
