# Day 3/100 — Design Patterns: Creational (Factory, Singleton, Builder)
**Date:** 2026-03-18
**Phase:** 1 — Engineering Foundations

---

## 📚 Lý thuyết cốt lõi

- **Factory Pattern** tách biệt việc *tạo object* khỏi *logic sử dụng* — caller không cần biết class nào được khởi tạo, chỉ cần biết interface
- **Abstract Factory** là "factory của factory" — tạo ra cả *họ* object liên quan mà không phụ thuộc vào implementation cụ thể
- **Singleton Pattern** đảm bảo một class chỉ có *một instance duy nhất* + cung cấp global access point; cần cẩn thận với threading & testability
- **Builder Pattern** tách biệt *construction* của complex object khỏi *representation* — đặc biệt hữu ích khi object có nhiều optional parameters
- **Creational patterns** giải quyết vấn đề cốt lõi: *"Ai có trách nhiệm tạo object?"* — câu hỏi Architect phải trả lời rõ ràng
- **Anti-pattern cần tránh:** `new` trực tiếp khắp nơi trong business logic — khi cần thay implementation, phải sửa rất nhiều chỗ
- **Khi nào dùng Factory vs Builder:** Factory khi logic tạo object phức tạp & cần runtime selection; Builder khi object có nhiều configuration steps theo thứ tự

## 💡 Ví dụ thực tế bằng TypeScript

### Factory Pattern — Notification Service

```typescript
// Interface chung
interface Notifier {
  send(message: string, recipient: string): Promise<void>;
}

// Các implementations
class EmailNotifier implements Notifier {
  async send(message: string, recipient: string) {
    console.log(`📧 Email to ${recipient}: ${message}`);
    // gọi email service
  }
}

class SlackNotifier implements Notifier {
  async send(message: string, recipient: string) {
    console.log(`💬 Slack to ${recipient}: ${message}`);
    // gọi Slack API
  }
}

class SMSNotifier implements Notifier {
  async send(message: string, recipient: string) {
    console.log(`📱 SMS to ${recipient}: ${message}`);
    // gọi SMS gateway
  }
}

// Factory — quyết định tạo loại notifier nào
type NotifierType = 'email' | 'slack' | 'sms';

class NotifierFactory {
  private static registry: Record<NotifierType, new () => Notifier> = {
    email: EmailNotifier,
    slack: SlackNotifier,
    sms: SMSNotifier,
  };

  static create(type: NotifierType): Notifier {
    const NotifierClass = this.registry[type];
    if (!NotifierClass) throw new Error(`Unknown notifier type: ${type}`);
    return new NotifierClass();
  }

  // Mở rộng không cần sửa Factory (OCP)
  static register(type: string, cls: new () => Notifier) {
    (this.registry as any)[type] = cls;
  }
}

// Usage — caller không quan tâm implementation
const channel = process.env.NOTIFICATION_CHANNEL as NotifierType || 'email';
const notifier = NotifierFactory.create(channel);
await notifier.send('Deployment succeeded! 🚀', 'truong@example.com');
```

### Builder Pattern — Query Builder

```typescript
interface QueryConfig {
  table: string;
  conditions: string[];
  fields: string[];
  limit?: number;
  offset?: number;
  orderBy?: string;
}

class QueryBuilder {
  private config: QueryConfig;

  constructor(table: string) {
    this.config = { table, conditions: [], fields: ['*'] };
  }

  select(...fields: string[]): this {
    this.config.fields = fields;
    return this; // fluent interface
  }

  where(condition: string): this {
    this.config.conditions.push(condition);
    return this;
  }

  limit(n: number): this {
    this.config.limit = n;
    return this;
  }

  offset(n: number): this {
    this.config.offset = n;
    return this;
  }

  orderBy(field: string): this {
    this.config.orderBy = field;
    return this;
  }

  build(): string {
    const { table, fields, conditions, limit, offset, orderBy } = this.config;
    let query = `SELECT ${fields.join(', ')} FROM ${table}`;
    if (conditions.length) query += ` WHERE ${conditions.join(' AND ')}`;
    if (orderBy) query += ` ORDER BY ${orderBy}`;
    if (limit) query += ` LIMIT ${limit}`;
    if (offset) query += ` OFFSET ${offset}`;
    return query;
  }
}

// Usage — step-by-step, readable, no giant constructor
const query = new QueryBuilder('users')
  .select('id', 'name', 'email')
  .where('active = true')
  .where('role = "admin"')
  .orderBy('created_at DESC')
  .limit(20)
  .offset(0)
  .build();

console.log(query);
// SELECT id, name, email FROM users WHERE active = true AND role = "admin" ORDER BY created_at DESC LIMIT 20 OFFSET 0
```

### Singleton Pattern — Config Service (thread-safe style)

```typescript
class ConfigService {
  private static instance: ConfigService | null = null;
  private config: Record<string, string> = {};

  private constructor() {
    // Load config once
    this.config = {
      DB_URL: process.env.DATABASE_URL || 'localhost:5432',
      API_KEY: process.env.API_KEY || '',
      ENV: process.env.NODE_ENV || 'development',
    };
    console.log('ConfigService initialized (once!)');
  }

  static getInstance(): ConfigService {
    if (!ConfigService.instance) {
      ConfigService.instance = new ConfigService();
    }
    return ConfigService.instance;
  }

  get(key: string): string {
    return this.config[key] ?? '';
  }

  // Dành cho testing — reset singleton
  static resetForTesting() {
    ConfigService.instance = null;
  }
}

// Dù gọi nhiều lần, chỉ init một lần
const cfg1 = ConfigService.getInstance();
const cfg2 = ConfigService.getInstance();
console.log(cfg1 === cfg2); // true
console.log(cfg1.get('ENV')); // development
```

## ❓ Câu hỏi phỏng vấn Architect level

1. **"Khi nào bạn sẽ chọn Abstract Factory thay vì Factory Method? Hãy cho ví dụ từ hệ thống bạn đã build."**
   *(Expect: candidate hiểu product families, ví dụ UI theming, multi-cloud SDK, database adapters)*

2. **"Singleton thường bị coi là anti-pattern trong modern software. Tại sao? Và khi nào nó vẫn là lựa chọn đúng?"**
   *(Expect: đề cập global state, testability issues, DI alternatives, nhưng cũng biết legitimate uses: logger, config, connection pool)*

3. **"Team bạn có một service khởi tạo rất phức tạp với 15+ config options. Junior dev đề xuất dùng Builder. Senior đề xuất dùng config object. Bạn sẽ quyết định thế nào và trình bày trade-offs?"**
   *(Expect: Builder → immutability, validation at build time, fluent API; Config object → simplicity, easy to serialize; context matters)*

## 💎 Insight của Architect

**"Creational patterns không phải về syntax — chúng là về *ownership of knowledge*."**

Khi bạn dùng `new SomeClass()` trực tiếp trong business logic, bạn đang nói: *"Module này biết và chịu trách nhiệm về cách tạo ra SomeClass."* Điều đó tạo ra coupling.

Architect tốt liên tục hỏi: **"Who should know HOW to create this?"**

- Config service biết cần DB adapter loại nào → Factory
- Order service cần build complex Order object từng bước → Builder  
- App chỉ cần một Logger duy nhất → Singleton (hoặc tốt hơn: DI container với singleton scope)

Khi code review, nếu thấy `new` bên trong business logic với logic phức tạp, đó là signal để refactor sang creational pattern. Không phải vì "pattern đẹp" mà vì **nó đặt trách nhiệm đúng chỗ**.

## 🧑‍💼 Day 3/100 — Leader Skills: Technical Writing (Viết Tài Liệu Kỹ Thuật)

### 🧠 Concept cốt lõi

1. **Viết cho người đọc, không phải cho bản thân** — Luôn hỏi: "Người đọc biết gì rồi? Họ cần quyết định gì sau khi đọc?" Tài liệu tốt giúp người đọc hành động, không chỉ "hiểu".
2. **Pyramid Principle** — Bắt đầu bằng kết luận/vấn đề chính, sau đó mới đưa ra lý do và chi tiết. Ngược lại với cách code (bottom-up), doc viết top-down.
3. **Phân loại tài liệu** — Tech Lead cần thành thục 4 loại: (a) Tutorial (hướng dẫn làm theo), (b) How-to Guide (giải quyết vấn đề cụ thể), (c) Explanation (giải thích tại sao), (d) Reference (tra cứu nhanh). Đừng trộn lẫn.
4. **One doc, one purpose** — Mỗi tài liệu chỉ phục vụ 1 mục tiêu rõ ràng. RFC để quyết định, ADR để ghi lại quyết định đã làm, README để onboard.
5. **Living docs > Perfect docs** — Tài liệu cập nhật 80% chính xác còn hơn tài liệu hoàn hảo nhưng lỗi thời 6 tháng. Gắn tài liệu gần với code (repo), không để riêng.

### 📖 Tình huống thực tế

**Scenario:** Team bạn vừa xong sprint xây dựng feature authentication mới (JWT + refresh token). Junior dev mới join hỏi: "Cái auth flow này hoạt động thế nào? Tại sao không dùng session?" — bạn giải thích miệng mất 20 phút, tuần sau lại có người hỏi y chang.

**Vấn đề:** Không có tài liệu, kiến thức chỉ tồn tại trong đầu người làm. Mỗi lần onboard tốn thời gian lặp lại. Khi bạn nghỉ phép, team bị block.

**Giải pháp của Tech Lead:** Sau khi giải thích lần đầu, viết ngay 1 ADR (Architecture Decision Record) ngắn: "Tại sao chọn JWT thay vì session?" + 1 diagram flow. Gắn link vào PR. Lần sau mọi người tự đọc.

### ✍️ Bài tập thực hành (15-20 phút)

**Task:** Viết 1 ADR (Architecture Decision Record) cho 1 quyết định kỹ thuật bạn đã làm gần đây (hoặc giả định).

Dùng template này, điền vào:
```markdown
# ADR-001: [Tên quyết định]

## Status: Accepted

## Context
[Vấn đề/tình huống dẫn đến quyết định này là gì?]

## Decision
[Quyết định cuối cùng là gì?]

## Consequences
### ✅ Pros
- ...
### ❌ Cons
- ...

## Alternatives Considered
- [Option A]: rejected vì...
- [Option B]: rejected vì...
```

**Gợi ý chủ đề nếu chưa có:** Tại sao chọn PostgreSQL thay vì MongoDB? Tại sao dùng monorepo? Tại sao không dùng microservices?

### 💎 Insight của Leader

> *"Tài liệu tốt nhất là tài liệu mà bạn không cần phải giải thích thêm sau khi người ta đọc xong."*

Nhiều senior dev nghĩ viết doc là công việc của tech writer hoặc "waste of time". Thực tế, khả năng viết tài liệu rõ ràng là **multiplier** — nó nhân lên impact của bạn lên toàn team. 1 ADR được viết tốt có thể tiết kiệm 10 cuộc họp trong 1 năm.

Khi bạn trở thành Tech Lead, người ta đánh giá bạn không chỉ qua code bạn viết, mà qua **clarity của thinking** bạn để lại trong tài liệu. Pyramid Principle đơn giản: bắt đầu bằng "Vì vậy, chúng ta nên làm X" — rồi mới giải thích tại sao.
