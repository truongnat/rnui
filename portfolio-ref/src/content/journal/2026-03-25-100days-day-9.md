Today was day 9 of my 100-day journey toward becoming a Software Architect and Tech Lead.

## What I Learned

### 🏗️ Architecture: DDD Bounded Contexts & Context Mapping

Today clicked something that's been fuzzy in my head for a while: the word "User" can mean completely different things depending on which part of your system you're in. In the Identity context, a `User` is about authentication — email, password hash, roles. In the Billing context, the same human being is a `Customer` with a payment plan and invoices. Mixing these concepts is where real systems start to break.

The **Bounded Context** is the explicit boundary inside which a domain model has one consistent meaning — and **Ubiquitous Language** lives *inside* that boundary. Cross the boundary, and you need translation, not assumption.

What made this concrete was understanding the Context Map patterns. **Anti-Corruption Layer (ACL)** stood out to me as the most important one. When I integrate with a legacy system or any upstream context I don't control, the ACL translates their model into my own language at the boundary — so their schema changes don't ripple through my domain logic. The TypeScript example of `IdentityServiceACL` translating a raw `User` into a `CustomerIdentity` that only exposes what Billing actually needs made this tangible.

The critical distinction: **Bounded Context ≠ Microservice**. A BC is a *design* decision about where your model's meaning holds. Whether it runs as a separate service is a *deployment* decision. I've seen teams conflate these and end up with either over-engineered microservices or a monolith that secretly has BC violations all over the place.

**Key takeaways:**
- Same word, different meaning in different contexts — that's by design, not a naming failure
- ACL is a firewall for domain integrity: change in Context A should never silently break Context B
- Shared Kernel requires tight team coordination; ACL trades some overhead for independence

### 🧑‍💼 Leadership: Writing Technical Documentation That Gets Read

The hard truth from today: if you write one 20-page doc for a CTO, a backend team, and a PM at the same time, nobody reads it fully. They each need something different.

**BLUF (Bottom Line Up Front)** was the framework that resonated most. Leaders don't scroll to find conclusions — the conclusion has to be the first thing they see. Everything after is proof and context for those who want it. This is a complete flip from how I typically write technical content, where I build up context before the punchline.

The **Micro-ADR** template is the one I'll start using immediately. Architecture Decision Records turn "why did we build it this way?" into a documented trail rather than tribal knowledge. The scenario about someone joining the team 6 months later and not knowing which decisions were intentional versus accidental — that's real. I've been that person. I've also been the person who couldn't explain a past decision because I never wrote it down.

**Key takeaways:**
- Audience segmentation: CTO needs "why", devs need "how", PM needs "when" — one doc can't serve all three equally
- Living docs beat dead docs: date + owner + status (Draft/Review/Final) make a doc trustworthy
- 1 diagram consistently outperforms 3 paragraphs of flow description

## How I Will Apply This

On the architecture side, I'm going to map out the Bounded Contexts in a project I'm currently working on. I've been letting the "product" concept from one module bleed into another, and I can already see the coupling it's creating. I'll draft a Context Map (even if it's just a Mermaid diagram) and identify the one integration point where I most need an ACL.

On the documentation side, I'm going to write an ADR for the last significant technical decision I made and store it in `docs/decisions/`. The goal isn't a perfect doc — it's starting the habit. I want future-me and future-teammates to be able to read the decision trail and understand not just *what* was built, but *why*.
